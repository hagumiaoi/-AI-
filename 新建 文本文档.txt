问题导入：我计划做一个项目，让ai来写论文。大概步骤：1.用户将需要参考的论文（知网上获得）放入input文件夹里 
2.启动服务，进入swagger界面输入需求（标题，还有要做什么）3.ai通过用户需求和input里的论文，进行大纲制定（写好标题 摘要建模等等），开始撰写论文，并在最后加上引用的论文   
我目前有想到的是fastapi，langchain，RAG

详细任务清单
第一阶段：环境搭建与文档预处理
文件监控模块：使用 watchdog 库监控 input 文件夹，当有新 PDF 放入时自动触发索引。
PDF 解析与清洗：
编写解析脚本，提取 PDF 的文本内容。
关键点：需要清洗页眉、页脚、参考文献列表（避免干扰检索）。
向量化（Embedding）：
将解析后的文本切片（Chunking），使用 text-embedding-3-small 或开源模型（如 bge-m3）存入向量数据库。
第二阶段：后端架构设计 (FastAPI)
Swagger API 定义：
定义 POST /generate 接口，参数包括：title（标题）、task_type（如：撰写综述、补充数据分析）、tone（学术风格要求）。
异步任务管理：
因为写论文很慢，接口不能直接返回全文。
流程：接口接收请求 -> 写入任务队列 -> 返回 task_id -> 客户端轮询或使用 WebSocket 获取实时进度。
第三阶段：AI 生成逻辑（LangChain 核心）
大纲生成器：
Prompt：基于用户标题 + 向量库中的相关论文，生成符合学术规范的结构（Introduction, Methodology, Results, Discussion 等）。
内容创作引擎（RAG）：
分块撰写：不要一次性生成全篇，采用“分段生成”模式。
每一段生成时，检索向量库中与当前段落主题相关的论文内容，作为 Context 给到 LLM。
引用追踪机制：
在生成的文字中标注 [1]，[2]。
难点：保持引用的一致性，需建立一个映射表（Mapping Table），将 [1] 准确对应到 input 文件夹中的某篇原始文件。
第四阶段：输出与格式化
格式渲染：
将 LLM 生成的 Markdown 文本转化为 PDF 或 Word (.docx) 格式（建议使用 python-docx 库）。
自动化参考文献列表：
根据文中引用的索引，自动在文末生成标准的参考文献格式（GB/T 7714）。
第五阶段：UI 与 交互优化
Swagger 界面调优：
利用 FastAPI 的 pydantic 定义 schema，让 Swagger 显示清晰的输入提示。
日志与状态回馈：
在控制台/日志中显示 AI 正在写的章节，方便 Debug。
三、 项目避坑指南（进阶建议）
RAG 的精度问题：知网下载的论文有很多图表和公式，简单的文本提取会失效。建议使用 MinerU 或 PaddleOCR 这种针对学术文档优化过的解析工具。
上下文长度限制：写论文需要超长上下文。如果遇到论文篇幅过长，建议使用 LongContext LLM（如 GPT-4o-128k）或在生成时引入“滑动窗口”机制。
避免“幻觉”：要求 AI 在撰写每一段话时，强制要求引用，并要求其必须基于检索到的片段（Context）进行改写，禁止 AI 凭空捏造数据。
引用管理：如果你希望生成非常专业的论文，建议引入 Zotero 的 API，通过 BibTeX 管理引用，而不是简单地让 AI 拼凑文字。
